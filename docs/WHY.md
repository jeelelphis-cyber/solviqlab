## Предисловие

Этот документ не технический.

**Bible** отвечает на вопрос «что построено».  
**ADR** отвечает на вопрос «почему принято конкретное решение».

Через три года этот документ позволит новому архитектору понять не только текущую систему, но и логику её развития — что резко снижает риск повторного появления уже однажды отвергнутых решений.

Каждый раздел — это момент, когда мы поняли что-то важное и изменили курс.

# Глава 0. Начало. Калькуляторная ферма.

**Дата: 2026-07-16**

Проект начинался как CALCO — платформа онлайн-калькуляторов.

Логика была простая: люди ищут "bmi calculator" в Google — 2.7 миллиона запросов в месяц. Добавим калькулятор, получим трафик. Добавим 36 калькуляторов, получим больше трафика.

Первая версия именно так и работала: 36 независимых инструментов, каждый оптимизирован под SEO, каждый существует сам по себе.

Пользователь приходил, считал BMI, уходил.

**Что было не так:**

Пользователь, который ищет BMI калькулятор, не ищет "математику". Он на пути к похудению, к здоровью, к изменению себя. Это намерение, не вычисление.

Платформа не понимала намерений. Она понимала только "запрос → результат".

**Идея, которую мы отвергли:** "Просто добавим больше калькуляторов. Больше калькуляторов = больше трафика = больше денег."

Почему отвергнута: ферма калькуляторов — это медиасайт с рекламой, не продукт. Трафик есть, ценности нет, удержания нет. Пользователь уходит сразу же после расчёта.

# Глава 1. Первый сдвиг. Intent Platform.

**Дата: 2026-07-23 (Sprint V3-08A)**

Вопрос, который изменил всё:

> "Почему пользователь считает BMI?"

Ответ: потому что он хочет похудеть. Или понять своё здоровье. Или подготовиться к визиту к врачу.

Это **намерение**. Intent.

BMI калькулятор — это не продукт. Это **точка входа** в Journey похудения.

Ключевое решение:

```
Старая модель:   Calculator → Result → (пользователь уходит)
Новая модель:    SEO Entry → Intent → Journey → Profile → Conversion
```

Каждый калькулятор — это начало пути, а не конец.

Мы переименовали CALCO в SolviqLab и переосмыслили всё.

Новое определение продукта: **Intent Cluster** — полная вертикаль вокруг одного намерения.

Для намерения "похудеть":
- 6 SEO entry points (BMI, TDEE, калории, и т.д.)
- Assessment — синтез всех данных
- Planner — план действий
- Tracker — отслеживание прогресса
- AI Coach — персональный советник

**Что это изменило:** калькуляторы перестали быть продуктом. Они стали входными точками.

# Глава 2. Память. ProfileEngine.

**Дата: 2026-07-23 (Sprint V3-05)**

После принятия Intent Platform концепции появился следующий вопрос:

> "Как платформа помнит, что уже знает о пользователе?"

Пользователь посчитал BMI (24.2, Normal). Потом TDEE (2400 kcal). Потом суточный дефицит калорий (300 kcal). Каждый результат — изолирован.

Нам нужна была **память**.

Решение: ProfileEngine. Каждый результат извлекает сигналы (HealthSignals) и увеличивает confidence в соответствующих доменах (weight, sleep, nutrition, и т.д.).

После трёх калькуляторов платформа знает: BMI норма, дефицит умеренный, TDEE средний. Confidence по weight domain = 67%.

**Первая идея (отвергнута):** хранить сырые результаты и каждый раз перечитывать их при отрисовке Dashboard.

Почему отвергнута: O(n) на каждый render. При 50 результатах — неприемлемо. Плюс: невозможно обнаруживать противоречия между разрозненными результатами.

**Принятое решение:** ProcessResult() извлекает сигналы немедленно при получении результата. Dashboard читает готовый Profile, не историю.

# Глава 3. Синтез. AssessmentEngine.

**Дата: 2026-07-23 (Sprint V3-08B/C)**

ProfileEngine решил проблему памяти. Но появилась новая проблема:

> "Профиль накопил данные. Но что с ними делать?"

Пользователь с BMI 27.2, дефицитом 200 kcal, 6 часами сна — что это означает? Это риск? Это норма? Что должно быть следующим шагом?

ProfileEngine знал ЧТО. Но не знал ЧТО ЭТО ЗНАЧИТ.

Нам нужен был слой синтеза.

Решение: AssessmentEngine. Принимает сигналы из ProfileEngine, прогоняет через ScoringRule DSL, строит narrative о состоянии пользователя.

**Ключевое осознание:** Assessment не измеряет. Он **синтезирует**.

BMI измеряет пользователь сам. Assessment берёт BMI + сон + дефицит калорий + стресс и говорит: "У вас умеренный риск плато из-за недосыпания. Рекомендую начать с нормализации сна перед интенсивными тренировками."

**Идея, которую мы отвергли:** встроить синтез прямо в Dashboard (читает данные и показывает интерпретацию).

Почему отвергнута: синтез — это бизнес-логика. В UI её быть не должно. Плюс: при добавлении Mobile/API нужно было бы дублировать логику.

# Глава 4. Источник правды. IntentState.

**Дата: 2026-07-23 (Sprint V3-10B)**

По мере роста системы возник тонкий но критичный вопрос:

> "Что является источником правды?"

ProfileEngine накапливал confidence. Казалось, ProfileEngine и есть источник правды.

Но потом мы поняли: **PersonalProfile — это проекция, не данные**.

Что настоящее? Настоящее — это сам факт того, что пользователь посчитал BMI. Сам факт, что он прошёл Assessment. Сам факт, что он создал план.

Profile — это интерпретация этих фактов. Интерпретация может меняться (улучшился алгоритм scoring), но факты остаются.

Решение: IntentState как главный платформенный объект.

```
Настоящее (в IntentState):
  - completedInstruments: InstrumentResult[]  ← факты
  - assessmentResults: AssessmentResult[]     ← синтез в момент T
  - activePlan: ActivePlan                    ← что пользователь делает
  - checkIns: CheckIn[]                       ← что происходит

Производное (PersonalProfile):
  - пересчитывается из InstrumentResult[]
  - при изменении алгоритма — пересчитывается вся история
  - никогда не хранится как источник правды
```

**Идея, которую мы отвергли:** ProfileEngine как центральный объект хранилища.

Почему отвергнута: если алгоритм scoring меняется, нет способа пересчитать историю. Плюс: ProfileEngine смешивал мутируемое состояние (confidence) с неизменяемыми фактами (результаты), создавая conceptual confusion.

# Глава 5. Вилка путей. StrategyEngine.

**Дата: 2026-07-23 (Sprint V3-09B)**

После IntentState стал следующий вопрос:

> "Assessment определил состояние пользователя. Дальше — Planner строит план. Но как Planner решает КАКОЙ план строить?"

Пользователь с BMI 28, умеренным стрессом, 5h сна — и тот же пользователь но с 8h сна и низким стрессом — нужны разные планы. Один → Conservative. Другой → Fast Track.

Если Planner сам принимает это решение — он становится слишком умным. Он смешивает "что строить" с "как строить". Это нарушение Single Responsibility.

Если Assessment передаёт решение о пути — Assessment знает о Planner, что тоже неправильно.

Решение: **StrategyEngine** — отдельный слой между Assessment и Planner.

```
Assessment → "Вот состояние мира" (факты и оценки)
Strategy   → "Вот оптимальный путь" (выбор из вариантов)
Planner    → "Вот план по выбранному пути" (исполнение)
```

**Ключевое доказательство:** один и тот же Assessment → два разных Planner без изменения ни Assessment, ни Planner. StrategyEngine — единственное место, где принимается это решение.

**Идея, которую мы отвергли:** "Пусть Assessment сразу содержит рекомендацию по стратегии".

Почему отвергнута: тогда Assessment знает о Planner. При добавлении третьего типа Planner нужно менять Assessment. Это backwards coupling.

# Глава 6. Изоляция бизнеса. PolicyEngine.

**Дата: 2026-07-23 (CEO Directive)**

По мере роста системы появился скрытый риск:

> "Бизнес-логика просочилась в движки."

В RecommendationEngine коэффициенты scoring негласно отдавали предпочтение платным продуктам. В JourneyEngine логика показа UnlockCard менялась в зависимости от "режима" платформы.

Каждый раз, когда нужно было изменить бизнес-приоритет (например, сфокусироваться на регистрациях вместо вовлечённости), нужно было менять код движков.

Это запрещено.

Решение: **PolicyEngine** как единственный слой с бизнес-логикой.

```
user_first  → Польза → Удержание → Простота
retention   → Польза → Вернуть завтра → Завершить Journey
roi         → Польза → LTV → Premium → AI Coach
growth      → Польза → Регистрация → Виральность
```

Переключить весь бизнес-режим платформы: `PLATFORM_POLICY=retention` в `.env`.

**Железное правило:** Policy не может добавить кандидата с `approved=false`. Safety/Medical флаги абсолютны. Policy только сортирует уже одобренных кандидатов.

**Идея, которую мы отвергли:** "Добавим mode параметр в каждый движок".

Почему отвергнута: 8 движков × 4 режима = 32 варианта поведения, разбросанных по всей кодовой базе. Невозможно тестировать и поддерживать.

# Глава 7. Пространство возможностей. Capability Layer.

**Дата: 2026-07-23 (Sprint V3-10B addendum)**

Когда мы думали о следующих продуктах, появился вопрос:

> "У нас есть weight cluster и sleep cluster. Что будет с 'Longevity'?"

Longevity — продукт о долголетии. Он использует: данные о весе, о сне, о питании, о стрессе. Он пересекает несколько кластеров.

Если кластер — верхний уровень архитектуры, то Longevity нужно залезть внутрь weight cluster и sleep cluster. Это нарушение boundaries.

Решение: **Capability Layer** над Cluster.

```
capabilities/
  health/           ← Capability (область знаний)
    weight/         ← Cluster (конкретное намерение)
    sleep/
    longevity/      ← Может использовать инструменты из weight/ и sleep/
  finance/
    budgeting/
```

Capability — это область знаний. Cluster — конкретное намерение пользователя внутри этой области.

**Идея, которую мы отвергли:** Cluster как плоский список на верхнем уровне.

Почему отвергнута: невозможно моделировать продукты, которые пересекают несколько областей намерений. Weight + Sleep + Nutrition = Longevity — это реальный продукт, который нельзя поместить ни в один из трёх кластеров по отдельности.

# Глава 8. Самоописание. Product Manifest.

**Дата: 2026-07-23 (Sprint V3-10B addendum)**

По мере роста платформы появилась проблема "невидимого coupling":

> "Чтобы добавить новый продукт, нужно отредактировать 4 файла вне папки продукта."

`INSTRUMENT_PROFILE_MAP` — знает о каждом продукте.  
`NEXT_STEP_DATA` — знает о каждом продукте.  
`CLUSTER_INSTRUMENTS` — знает о каждом продукте.  
`journey/config.ts` — знает о каждом продукте.

Это означало: каждый движок содержит список всех продуктов. Добавление продукта = изменение 4 движков. Это прямо противоположно P-13 (One Product Rule).

Решение: **Product Manifest** + **CapabilityCatalog**.

```typescript
// Продукт описывает себя:
export default defineInstrument({
  requires: ['weight', 'height'],
  provides: ['bmi_score', 'bmi_category'],
  emits: ['solviqlab:result'],
  profileSignals: ['bmi', 'bmi_fitness_proxy'],
  journeyStep: { journeyId: 'weight-management', position: 1 },
})
```

CapabilityCatalog строится из манифестов автоматически при старте. Движки читают каталог. Никаких хардкодов.

**P-13 теперь проверяем:** `git diff --name-only` при добавлении нового продукта должен показывать только файлы внутри папки продукта.

# Глава 9. Финальный слой. Event-Driven Runtime.

**Дата: 2026-07-23 (CEO Directive V3-10D)**

Даже с Policy Engine, Capability Layer и Product Manifest оставалась одна проблема:

> "Как продукт общается с платформой?"

AssessmentClient читал ProfileEngine напрямую.  
CalculatorClient диспатчил CustomEvent — но что происходило дальше, было implicit.

Если завтра появляется Mobile-приложение — каждый продукт нужно переписать, чтобы он знал о мобильном контексте.

Если появляется Telegram-бот — снова переписывать.

Решение: формализовать **Event Bus** как единственный канал коммуникации.

```
Продукт знает только:
  execute() → emit(result)

Всё остальное — ответственность платформы:
  → UserEngine сохраняет
  → ProfileEngine пересчитывает
  → AssessmentEngine проверяет trigger
  → StrategyEngine выбирает путь
  → PolicyEngine сортирует
  → RecommendationEngine обновляет
  → Dashboard реагирует
```

Транспорт EventBus меняется:
- Browser → CustomEvent API
- Mobile → DeviceEventEmitter
- Server → Redis Streams
- AI Agent → gRPC stream

**Продукты не меняются. Контракты не меняются.**

Это делает платформу channel-agnostic с первого дня.

# Что было отвергнуто: полный список

| Идея | Почему отвергнута |
|------|-------------------|
| Ферма калькуляторов | Медиасайт с рекламой, не продукт. Нет удержания. |
| Движки вызывают друг друга напрямую | Spaghetti dependencies. Невозможно тестировать. |
| ProfileEngine как единственный источник правды | Путает мутируемые данные с историей фактов. |
| Синтез в UI (Dashboard читает и интерпретирует) | Бизнес-логика в презентационном слое. Нельзя переиспользовать. |
| Бизнес-режим как параметр каждого движка | 8 движков × 4 режима = 32 разрозненных варианта. Непригодно. |
| Cluster как верхний уровень архитектуры | Невозможно cross-cluster продукты (Longevity). |
| Реестр продуктов как хардкод в движках | Добавление продукта меняет движки. Нарушает P-13. |
| Продукты вызывают платформу напрямую | Coupling к конкретной платформе (Web). Нельзя добавить Mobile. |
| `mode` параметр в каждом движке для бизнес-логики | Скрытая логика в движках. Нельзя переключать без кода. |

# Итоговая эволюция

```
День 1:   36 калькуляторов. Каждый сам по себе.
           → Проблема: нет памяти, нет пути, нет ценности.

День 7:   Intent Platform. Journey Engine.
           → Решение: каждый вход = начало пути.

День 7:   ProfileEngine. Память.
           → Решение: накопление сигналов, confidence по доменам.

День 7:   AssessmentEngine. Синтез.
           → Решение: данные → смысл → рекомендация.

День 7:   IntentState. Настоящий источник правды.
           → Решение: факты неизменны. Profile — производная.

День 7:   StrategyEngine. Вилка путей.
           → Решение: Assessment → Strategy → Planner. Независимо.

День 7:   PolicyEngine. Изоляция бизнеса.
           → Решение: один env var = весь бизнес-режим.

День 7:   Capability Layer. Пространство роста.
           → Решение: cross-cluster продукты без нарушения границ.

День 7:   Product Manifest. Самоописание.
           → Решение: продукт = 1 файл. Движки не знают о продуктах.

День 7:   Event-Driven Runtime. Финальный слой.
           → Решение: продукт = execute() + emit(). Всё остальное — платформа.
```

Весь путь пройден за 7 дней.

Не потому что мы были быстрые. А потому что мы принимали правильные вопросы раньше, чем писали код.

**Самый ценный навык в архитектуре — знать, когда остановиться и спросить: "А что здесь настоящее?"**
